# Context Layering

## 1. 目标

当前项目已经有：

- 根 agent
- 一组真实可调用的工具
- session 历史
- todo 状态

但这些信息现在还没有一个明确的“上下文分层模型”。

这一步的目标不是一次把完整上下文工程做完，而是先立一个最小 context builder，让系统知道不同类型的信息应该放在哪一层，以及这些层在运行时如何被拼装。

## 2. 最小三层

当前阶段至少分成三层：

### 2.1 第一层：稳定层

放稳定、低频变化、必须长期成立的内容：

- 最小主 system prompt
- 最小工具使用规则

这层的特点是：

- 不依赖当前仓库文件内容
- 不依赖某一轮用户输入
- 不应该和 session 历史混在一起

### 2.2 第二层：仓库规则层

放当前仓库本地规则文档：

- `code_law.md`

这层的特点是：

- 属于仓库本地约束，而不是 agent 全局常量
- 内容相对稳定，但可能随仓库演进而改变
- 当前阶段按“项目根目录固定文件”读取

### 2.3 第三层：运行层
放当前会话与当前轮输入有关的动态内容：
- session 历史里的 user / assistant / tool 消息
- 本轮 user 输入

这层的特点是：
- 动态变化最快
- 与 session 强相关
- 后续 summary、截断、@file reminder、todo recap 等都应优先接在这一层附近，而不是塞回 L1/L2

## 3. 每层分别放什么

### 3.1 L1 最小 system prompt

当前阶段只覆盖已经真实存在的能力：

- 你是本地代码助手
- 回答本地代码问题前先用证据工具
- 现有工具有哪些
- 修改文件前要先读
- `TodoWrite` 适用于多步骤 coding 任务
- `Bash` 只用于非交互本地命令，不替代只读工具

### 3.2 L1 最小工具规则

当前只覆盖真实已落地工具：

- `LS`
- `Glob`
- `Grep`
- `Read`
- `Edit`
- `Write`
- `TodoWrite`
- `Bash`

不为未实现能力写规则：

- 不写 tracing 规则
- 不写 summary 压缩规则
- 不写 `@file` reminder 规则
- 不写复杂 truncation 规则

### 3.3 L2 仓库规则文档

当前只有：

- 项目根目录的 `code_law.md`

未来如果需要，可扩成：

- `AGENTS.md`
- `CLAUDE.md`
- 子目录规则文件

但这一步不做多文件查找和合并。

### 3.4 L3 运行层

当前最小包含：

- SDK session 已保存的历史消息
- 当前 user 输入

这一步先不把 todo recap、summary 和 @file reminder 注入进去。

## 4. 哪些内容必须稳定，哪些内容是动态的

### 4.1 必须保持稳定

- L1 的 system prompt
- L1 的工具规则结构
- L2 的注入位置
- 各层的边界定义

这些内容一旦随每轮输入频繁变化，模型会更难形成稳定行为。

### 4.2 可以动态变化

- `code_law.md` 的文件内容
- session 历史条目
- 当前 user 输入

未来还会有更多动态内容，但这一版先不接。

## 5. 为什么现在就要落最小 system prompt 和最小工具规则

原因很直接：

- 当前项目已经不是“只有一个空 agent”
- 已经有多种工具和不同职责边界
- 如果不先把稳定层立住，工具使用规则会继续散落在 `agent_factory.py` 或临时字符串里
- 后面一旦开始做 summary、@file、truncation、todo recap，就会更难区分哪些是稳定前缀，哪些是动态输入

所以这一步先做最小版本，不追求完整，只追求：

- 有稳定层
- 有仓库规则层
- 有运行层
- 三层边界清楚

## 6. 为什么不要把三层直接混成一大段字符串

如果一开始就把所有内容混成一个大字符串，会马上失去这几个能力：

- 看不清每一层来自哪里
- 后续很难只替换其中一层
- 测试时很难断言某一层是否正确注入
- 后续接 summary / recap / reminder 时容易彼此污染

因此，这一步的要求不是“永远不做字符串拼接”，而是：

- 内部必须先保留结构化分层
- 只有在贴近 SDK 调用边界时，才做最小适配

## 7. 当前阶段刻意不做什么

这一步先不做：

- 历史压缩
- summary 生成
- 工具输出截断
- `@file` 预处理
- todo recap 注入
- 噪声控制策略
- 多规则文件搜索
- 子目录规则继承

这些能力后续都可能建立在本次的最小 context builder 之上，但这一步不提前实现。

## 8. 当前建议的最小实现形态

### 8.1 结构

- `src/context/context_builder.py`
  - 负责构造三层对象
  - 负责把三层适配成 agent instructions 与 runner input
- `code_law.md`
  - 作为当前唯一仓库规则文档

### 8.2 输出

builder 当前最小应能产出：

- 一份结构化的 layer bundle
- 一份给 `Agent.instructions` 的稳定层字符串
- 一份给 `Runner.run_streamed(..., input=...)` 的输入项列表

这样既能保证内部边界清晰，也能和当前 SDK 运行方式兼容。
