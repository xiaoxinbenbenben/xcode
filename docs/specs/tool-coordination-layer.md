# Tool Coordination Layer

## 1. 为什么当前项目仍然需要一层工具协调层

当前项目已经把工具注册交给 OpenAI Agents SDK，但这并不意味着“工具层什么都不用管”。

我们仍然需要一层**轻量协调层**，原因有三点：

- 需要集中列出当前启用的工具，避免工具清单散落在多个运行时文件里
- 需要一个共享挂载点，放工作区边界、统一响应协议、最小共享状态等公共能力
- 需要为后续步骤留出稳定扩展点，例如乐观锁、Todo 状态、Bash 安全规则、输出截断

也就是说，这层不是拿来重建一套大而全的 registry，而是让工具系统有一个清晰、稳定、可扩展的落点。

## 2. 当前轻量协调层的最小形态

当前项目里的轻量协调层主要由两部分组成：

- `src/tools/__init__.py`
  - 负责集中导出工具分组
  - 负责组装 `AGENT_TOOLS`
- `src/tools/common.py`
  - 负责放所有工具共享的基础 helper
  - 包括工作区路径解析、协议上下文构造、统一错误映射、最小统计信息、文件快照等

这两层加在一起，就足够支撑当前的只读工具、编辑链路、Todo 工具和后续 Bash 工具。

## 3. 这层应该保留哪些职责

当前阶段，这层应该保留这些职责：

- 组织当前可用工具清单
- 提供共享 helper，避免每个工具重复写工作区边界校验
- 统一构造协议层 `context`、`stats` 和结构化错误
- 为运行时共享状态提供轻量挂载点
- 为后续新工具提供统一接入位置

## 4. 哪些旧 registry 职责不应该照搬

旧项目中一些 registry 职责在新项目里不应该直接搬过来：

- 不重建自定义 schema 汇总系统
- 不重建一套完整的工具执行框架
- 不在这一阶段引入熔断器
- 不把所有工具调用都包进一个大而重的统一入口

原因是：

- Agents SDK 已经能从 `function_tool` 自动生成 schema
- `Agent(tools=[...])` 已经能完成注册
- 当前真正需要自控的是安全边界、返回协议和共享状态，而不是再造一层工具平台

## 5. 这层如何服务后续工具

### 5.1 只读工具

- 共享工作区路径解析
- 共享协议层包装
- 共享错误映射

### 5.2 编辑链路

- 共享文件快照结构
- 共享工作区边界
- 共享乐观锁错误包装

### 5.3 Todo 工具

- 通过 runtime context 共享当前 todo 状态和完成态归档信息
- 通过统一协议返回结构化结果

### 5.4 Bash 工具

- 共享工作区目录解析
- 共享协议封装和错误映射
- 在同一工具清单里挂载到 agent 上

### 5.5 后续输出截断

- 若后续引入统一 Observation 截断或文件落盘策略，优先接在共享 helper 或工具分组层
- 不要把截断逻辑分散到每个工具内部各写一套

## 6. 当前边界

这层当前仍然是轻量协调层，而不是重型 registry：

- 工具仍然是普通 Python 函数
- schema 仍然由 SDK 自动生成
- agent 仍然直接持有 `AGENT_TOOLS`
- 共享逻辑只收敛到最小 helper，不做多余抽象

这保证了后续每加一个工具，都只是“在现有结构上增量接入”，而不是“再搭一套新框架”。
